# Delivery. ADR-promote — решения в канон

![Два канона: ADR-promote и doc-promote](assets/canons_adr_promote_docs_promote.png)

```
/spec-ship:adr-promote
```

## Что это

Превращает кандидата-решение (`adr-entry-*.json`, статус Proposed), которое build предложил по ходу реализации, в канонический **ADR** — Architecture Decision Record: markdown-документ «почему мы решили так» со сквозным номером, зарегистрированный в индексе.

Это **ручной шаг maintainer'а** после мёржа MR. Агент исполняет конвертацию, человек инициирует и апрувит — сам себе в канон агент писать не может.

## Зачем

Решения, принятые в чате, умирают с чатом. Через полгода кто-то «исправит» неочевидный код обратно — потому что не знал, что так сделано намеренно. ADR-канон решает это: каждое заметное решение — отдельный документ с контекстом, последствиями и отвергнутыми альтернативами. А скиллы пайплайна читают этот канон при каждой новой фиче и не дают агенту молча нарушить принятое.

## Предусловия (все обязательны)

- ReviewReport — APPROVED, проверка ADR-нарушений пройдена
- MR апрувнут maintainer'ом

Не выполнено — не промоутить.

## Как проходит

1. Кандидаты берутся из BuildReport (`adr_entries[]`).
2. Номер: максимальный в `.ship/docs/adr/` + 1. Нумерация сквозная, номера устаревших ADR не переиспользуются.
3. Область (Area) выводится из затронутых файлов — по ней потом фильтруется выборка.
4. JSON-кандидат конвертируется в markdown: Context, Decision, Consequences, Considered Options.
5. Строка добавляется в `.ship/docs/adr/INDEX.md` (статус Accepted, Area).
6. Кандидат в pipeline помечается `promoted` — остаётся как след трассируемости.

Если промоутимое решение противоречит уже принятому ADR — это не промоушен, а конфликт: стоп и явное разрешение через протокол ADR-CONFLICT.

## Почему канон не разрастается

Скиллы никогда не читают все ADR подряд — только `INDEX.md`, фильтр по статусу Accepted и пересечению Area с текущей фичей, и затем тела только совпавших. Сто ADR в каноне → в контекст агента попадает один индекс и 2–3 тела. Устаревшие (Expired) выпадают из выборки автоматически.

## Что от вас потребуется

- Инициировать после мёржа MR
- Подтвердить Area, если она неоднозначна
- Апрувнуть результат
