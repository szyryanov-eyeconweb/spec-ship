# Fan-out режим build — дизайн

> **Статус: реализовано 2026-06-25.** Внесено в `skills/decompose/SKILL.md` (раздел 3.5 + поле `fan_out` в схеме TaskSpec) и `skills/build/SKILL.md` (под-трек FAN-OUT, Phase A/A.5/B/C). Этот док — исходный дизайн-рационал, канон механики — в скиллах.

## Решения (зафиксировано 2026-06-25)

- **Ось разбиения — по РОЛИ слоя, не по путям проекта.** Пути в разных проектах
  разные; скиллы описывают роль, проект мапит на свои каталоги. Три роли:
  1. **Entry** — всё, принимающее извне: GraphQL/Kafka/RPC/Temporal-входы,
     резолверы, контроллеры, консьюмеры, активности.
  2. **Application** — use-cases, Handler, ПОРТЫ (интерфейсы) и DTO. Здесь рождается
     контракт, который реализуют остальные.
  3. **Contract-impl** — реализации портов Application (`implements <PortИзApplication>`):
     запросы к БД, репозитории, внешние клиенты.
- **RED — вариант 1.** ВСЕ тесты пишутся ДО фан-аута (Phase A.5). Тесты через
  публичный интерфейс = контракт поведения, проверяют стык слоёв. Phase B-агенты
  гонят свой слой под зелёные.

## Суть

Внутри ОДНОЙ задачи build реализует несколько слоёв ПАРАЛЛЕЛЬНО разными
сабагентами. Условие: контракт между слоями заморожен ДО фан-аута. Слои зависят
не от кода друг друга, а от контракта (интерфейс + DTO) — зафиксировали контракт,
дальше реализации независимы.

## Фазы

### Phase A — Contract (последовательно, основная сессия)
Пишет ТОЛЬКО контрактные артефакты — то, на что смотрят все слои:
- порты-интерфейсы из слоя Application
- DTO с сигнатурами (поля, типы) — без тел методов реализаций
- НЕ реализацию

Источник скелета — `shape.intermediate_structures` + канон-док паттерна, если он
уже есть (паттерн-док в `.ship/docs/workflows/` описывает слои и их контракты —
Phase A берёт готовый список интерфейсов/DTO, не изобретает).

Заморозка: контракт-артефакты read-only для Phase B.

### Phase A.5 — Tests (последовательно, RED)
`ship-red` пишет ВСЕ тесты задачи по `test_scenarios` — через публичный интерфейс,
без моков доменных внутренностей. Тесты ассертят сквозное поведение (стык слоёв),
не отдельный слой. Подтвердить ALL RED. Тесты становятся общим контрактом
поведения для всех Phase B-агентов.

### Phase B — Fan-out (параллельно, N сабагентов `ship-green`, каждый в своём worktree)
По одному сабагенту на роль-слой, каждый в отдельном git worktree. Каждый:
- видит контракт + общую землю Phase A (read-only) + красные тесты
- пишет ТОЛЬКО свой `layer.files_to_change` (граница записи — как обычная изоляция green)
- гонит suite, делает зелёным то, что покрывает его слой

Worktree даёт две вещи поверх инварианта непересечения путей:
- физический барьер ФС — слой не затрёт чужой файл даже по ошибке (страховка, не замена инварианта);
- детектор вылазок — diff worktree показывает запись вне границ слоя = незамеченная общая
  земля → стоп, в decompose дополнить `shared_paths`.

### Phase C — Merge + Integration (последовательно, основная сессия)
1. Merge worktree-слоёв: слой-файлы непересекающиеся → union; контракт/общая земля одинаковы
   во всех worktree → тривиально. Конфликт merge = пропущенная вылазка → стоп, в decompose.
2. Регенерация генерируемой земли (схема из аннотаций, DI-кеш, autoload) из объединённого дерева.
3. Полный прогон suite, ревью по полному результату. Стык не сошёлся (слой A вернул не то, что
   слой B ждал) — контракт был неточен, см. «незрелый контракт».

## Контракт fan-out в TaskSpec (расширение схемы)

```jsonc
"fan_out": {
  "enabled": true,
  "contract_paths": [ "<пути контракт-артефактов: порты + DTO>" ],
  "shared_paths":   [ "<общая земля: DI/реестр/схема; Phase A пишет за все слои>" ],
  "layers": [
    { "role": "application",   "files_to_change": ["<пути слоя>"] },
    { "role": "contract-impl", "files_to_change": ["<пути слоя>"] },
    { "role": "entry",         "files_to_change": ["<пути слоя>"] }
  ]
}
```
- `role` — из {entry, application, contract-impl}. Пути — проектные.
- `shared_paths` — общая земля (опционально, [] если нет): файлы, что трогают несколько
  слоёв, но не контракт — DI-реестр, регистрация сервисов/маршрутов, явные записи схемы.
  Phase A пишет их ЦЕЛИКОМ за все слои (регистрация вперёд реализаций). Генерируемую землю
  (схема из аннотаций, DI-кеш) сюда НЕ класть — регенерируется в Phase C.
- Инвариант 1: `files_to_change` слоёв попарно НЕ пересекаются.
- Инвариант 2: `contract_paths` ∩ любой `layer.files_to_change` = пусто.
- Инвариант 3: `shared_paths` ∩ любой `layer.files_to_change` = пусто.
- decompose валидирует все три; нарушение → не fan_out, обычный build.

## Условия применимости (decompose решает)

`fan_out.enabled = true` ТОЛЬКО если ВСЕ:
1. задача разбивается на ≥2 слоя-роли с непересекающимися путями;
2. контракт между слоями можно зафиксировать в Phase A (известен из shape/канона);
3. `trust_zone != CRITICAL` (критичное — Dev сам, без фан-аута).

Иначе — обычный последовательный build. Fan-out — оптимизация, не дефолт.

## Незрелый контракт (главный риск)

Phase B-агент понял, что контракт неполон (порту нужен метод, в DTO не хватает поля):
→ STOP всего фан-аута — НЕ чинить контракт внутри Phase B (гонка: двое правят порт)
→ вернуть в Phase A, дополнить контракт, заморозить заново
→ перезапустить Phase B
Это и есть «пересекающиеся данные»: пересекаются ТОЛЬКО при незрелом контракте.
Зрелость контракта = предусловие фан-аута, поэтому fan_out выгоден, когда контракт
известен заранее (есть паттерн-канон).

## Общая земля (второй риск — подтверждён пилотом)

Пилот CPA-dashboard (клон RevShare) показал: GREEN-агент тронул файлы вне любого слоя —
`config/.../graphql_generator.yaml`, генерируемую GraphQL-схему, общий `SubIdFilterMapper`.
Это не контракт и не код слоя — это «общая земля»: DI-реестр, регистрация, схема. При
параллели несколько слоёв дописывают свои записи в один файл → конфликт записи (runtime)
или конфликт merge (если в worktree).

Три типа конфликта, кто чинит:
| Конфликт | Чинит |
|---|---|
| race одного файла | инвариант 1 (непересечение) + worktree как страховка |
| общая земля (config/DI/схема) | `shared_paths` в Phase A + worktree-детектор вылазок |
| незрелый контракт (семантика) | Phase C, рестарт Phase A |

worktree изолирует ФС, но конфликт общей земли НЕ убирает — переносит с runtime-race на
merge-time. Настоящее решение: декларативную землю (явные записи) пишет Phase A целиком за
все слои (`shared_paths`); генерируемую (схема из аннотаций, DI-кеш) — регенерация в Phase C.
worktree-diff служит детектором: вылазка за `files_to_change ∪ contract_paths ∪ shared_paths`
= незамеченная земля → стоп, дополнить `shared_paths`, перезапуск.

## Выигрыш / цена

- Выигрыш: wall-clock Phase B = max(слой), не sum(слои). 3 слоя → до ~3× на этапе.
- Цена: последовательные Phase A + A.5 + риск рестарта при незрелом контракте.
- Окупается: ≥3 нетривиальных слоя + контракт известен заранее (канон есть).
- НЕ окупается: правка одного класса, 1–2 слоя, незрелый контракт.

## Что меняется в скиллах

- **decompose**: определять fan_out-able по трём условиям; нарезать `layers[]` по
  ролям (entry/application/contract-impl) + `contract_paths`; валидировать
  непересечение. В тексте — РОЛИ, не пути конкретного проекта.
- **build**: под-трек для `fan_out.enabled` — Phase A (контракт, сессия) → A.5
  (RED все тесты) → Phase B (parallel `ship-green` по слоям, изоляция
  `files_to_change`) → Phase C (интеграция). Незрелый контракт → стоп+рестарт A.
- **ship-green**: правок не нужно — уже изолирован правами; fan-out передаёт каждому
  его `layer.files_to_change` + `contract_paths` как read-only.
- **review**: без изменений — ревьюит полный результат.

## На будущее (не сейчас)

Вторая ось — независимые ветки внутри слоя (напр. варианты по типу актора). Можно
комбинировать роль × вариант, но риск гонки растёт. Старт — только три роли-слоя.
