# Запуск всей цепочки одной командой: run

![run: вся цепочка одним запуском, стоп только на ваших гейтах](assets/run.png)

```
/spec-ship:run Фильтрация транзакций по меткам
```

## Что это

`run` — дирижёр. Вместо того чтобы вызывать пять команд по очереди (`survey` → `shape-doc` → `decompose` → `build` → `review`), вы запускаете одну, и агент сам прогоняет всю цепочку. Останавливается он только там, где решение действительно ваше.

Сам `run` работу не делает — он по очереди вызывает те же скиллы этапов и управляет переходами между ними. Механика каждого этапа не меняется.

## Зачем

Полный пайплайн — это много этапов, и при ручном вызове каждый требует вашего внимания на старте: ввести команду, дождаться, прочитать, ввести следующую. Для одиночной мелкой задачи это перебор. Но для настоящей фичи на несколько задач набегает пять переключений контекста, между которыми агент простаивает в ожидании вас.

`run` убирает именно это. Важно понимать, **что именно экономится**:

- **Не экономятся** токены и машинное время — этапы всё равно выполняются полностью.
- **Экономятся** ваши ручные вызовы и переключения: один запуск вместо пяти, и агент не ждёт вас на автономных участках. Пока вы заняты другим, рутинные задачи собираются и проверяются сами; уведомление в Telegram придёт, только когда понадобитесь вы.

## Где агент останавливается, а где катит сам

Главное правило: **гейты участия человека все на месте**. `run` ничего не «протаскивает» мимо вас тайком.

| Этап | Поведение |
|---|---|
| survey | автономно (только наблюдает) |
| shape-doc: интервью | **стоп**, если есть блокирующие вопросы |
| shape-doc: апрув спеки | **стоп** на ваш апрув |
| decompose: апрув разбивки | **стоп** на ваш апрув |
| build: ROUTINE-задача | автономно (RED → GREEN) |
| build: LOGIC-задача | **стоп**: вы проектируете решение (шейп-сессия) |
| build: CRITICAL-задача | **стоп**: код пишете вы |
| любая эскалация | **стоп** по протоколу этапа |
| review: APPROVED | автономно дальше |
| review: NEEDS_WORK | агент чинит сам и проверяет заново — вас не дёргает |
| review: ESCALATE | **стоп**: к вам с контекстом |

То есть вы подключаетесь на принятии решений (апрув, проектирование, разбор эскалаций), а механическую работу и рутинные задачи агент делает без вас.

## Параллельность задач

Заблокированная задача не останавливает весь прогон. Пока одна LOGIC-задача ждёт, что вы её спроектируете, независимые ROUTINE-задачи уже собираются и проходят ревью. Стоп всего прогона — только если работать буквально не над чем без вашего решения.

Это и есть выигрыш: вы возвращаетесь к одной точке принятия решения, а к этому моменту половина фичи уже готова.

## Флаги автономности (по желанию)

По умолчанию все гейты включены. Если хотите ещё меньше пауз — снимите низкорисковые гейты:

| Флаг | Что снимает | Условие безопасности |
|---|---|---|
| `--auto-shape` | авто-апрув спеки | только если в ней 0 блокирующих вопросов |
| `--auto-decompose` | авто-апрув разбивки | только если все задачи ROUTINE |
| `--auto` | оба | те же условия |

```
/spec-ship:run --auto Импорт справочника партнёров из CSV
```

Что флаги **не** снимают никогда: блокирующие вопросы интервью, шейп LOGIC, CRITICAL-задачи, эскалации, review ESCALATE. Эти точки принципиально требуют человека, и обойти их нельзя by design.

Хороший кейс для `--auto` — ночной прогон рутины: вечером запустили, утром читаете сводку, Telegram разбудил бы только при настоящей проблеме.

## Возобновление

Если прогон остановился на гейте — ответьте/апрувните/зашейпьте и продолжите:

```
/spec-ship:run --resume
```

Агент прочитает, что уже сделано (по артефактам в `.ship/pipeline/{slug}/`), и продолжит с первой незавершённой точки. Готовое не переделывается.

## Что run не трогает

- **Delivery-этапы** (`adr-promote`, `doc-promote-feature`) — они после мёржа MR, это отдельный ручной шаг maintainer'а. В автопрогон не входят.
- **Одиночные задачи** — если задача одна и понятная, дешевле прямой `/spec-ship:build`, чем весь `run`.

## Когда что брать

| Ситуация | Команда |
|---|---|
| фича на 3–15 задач, хочется один запуск | `/spec-ship:run` |
| ночной прогон рутины | `/spec-ship:run --auto` |
| одна понятная задача | `/spec-ship:build` напрямую |
| только разведка или только спека | `/spec-ship:survey` / `/spec-ship:shape-doc` |
| опечатка, бамп версии | без пайплайна |
