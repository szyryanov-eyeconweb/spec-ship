# Delivery. Doc-promote — поведение в канон

![Два канона: ADR-promote и doc-promote](assets/canons_adr_promote_docs_promote.png)

```
/spec-ship:doc-promote                          # режим feature (после мёржа фичи)
/spec-ship:doc-promote --backfill survey-NNNN   # режим backfill (из существующего кода)
```

## Что это

Переносит знание о **поведении системы** в долгоживущие workflow-доки `.ship/docs/workflows/` с индексом и жизненным циклом.

Зеркало adr-promote, но для другого вида знания:

| | ADR-канон | Workflow-канон |
|---|---|---|
| Хранит | «почему решили так» | «как система себя ведёт» |
| Главный читатель | shape-doc, decompose, build, review | survey следующей фичи |

## Два режима

| Режим | Когда | Источник | Предусловия |
|---|---|---|---|
| **feature** (по умолчанию) | после мёржа фичи | survey + BusinessDoc | ревью APPROVED, MR смёржен |
| **backfill** | заранее, из уже написанного кода | только survey | survey есть и описывает код текущей ветки |

Оба — **ручной шаг maintainer'а**. Агент исполняет конвертацию, человек инициирует.

## Зачем

Без канона каждый survey по одной области трассирует код заново — это дорого. С каноном survey сначала читает готовый док (только актуальные, только по нужной области через индекс) и трассирует лишь расхождения. Каждая следующая фича в области дешевле предыдущей.

## Режим feature

Запускается после мёржа фичи — описывает поведение, которое РЕАЛЬНО в ветке, а не намерение.

1. **Предусловия:** ревью APPROVED, MR смёржен.
2. **Источник:** survey (поведение ДО фичи) + BusinessDoc (`feature.workflow`, критерии — поведение ПОСЛЕ). Итог описывает состояние ПОСЛЕ мёржа: workflow из bd замещают изменённые ими наблюдения survey, незатронутые переносятся как есть.
3. Если `feature.workflow` записан рефакторинг-формой `{старый} --шаг--> {новый}` — левая часть указывает замещаемый workflow канона, в документ попадает правая.

## Режим backfill

Наполняет канон поведением **уже написанного** кода — заранее, до того как по этой области пойдёт новая фича. Цель: типовые паттерны (graphql-эндпоинт, отчёт, колонка, консьюмер) описать один раз, чтобы будущие фичи стартовали не с нуля.

1. **Предусловие:** есть `survey-*.json`, трассирующий код текущей ветки. ReviewReport и MR-гейт НЕ требуются — фичи нет, есть существующий код.
2. **Источник:** только survey. bd отсутствует, поэтому `observed_workflows` survey — это и есть финальное поведение (не «AS-IS до фичи»).
3. **Помета:** в поле Source документа пишется `backfill (<survey-id>)` — видно, что канон извлечён из готового кода, а не прошёл пайплайн.
4. Когда фича позже изменит этот код-путь, doc-promote в режиме feature сделает amend и заместит backfill-наблюдение точным TO-BE.

Типовой backfill-заход: `/spec-ship:survey <эталонный символ>` → `/spec-ship:doc-promote --backfill <survey-id>`. Для паттерна полезно разводить два уровня: документ-**паттерн** (шаблон слоёв, переиспользуемый) и документ-**экземпляр** (конкретная фича) — они ссылаются друг на друга.

## Общая часть (оба режима)

1. **Целевой документ.** Область (Area) выводится из затронутых файлов. Док по этому код-пути уже есть → amend (обновить), нет → create.
2. **Сборка.** Формат: Definitions (структуры, таблицы, события), Workflows (стрелочная нотация + пояснения), словарь имён состояний. Стрелочные строки копируются из артефактов, не пересочиняются. Документ читается без открытия кода.
3. **Индекс.** Строка в `.ship/docs/workflows/INDEX.md`: документ, статус, область, источник, дата. Если фича задела поведение в **других** доках — им проставляется честный статус: `partially-outdated` или `superseded`. Молча лгущий документ — худший исход.

## Жизненный цикл документов

| Статус | Значение |
|---|---|
| `current` | авторитетно описывает поведение; survey ему доверяет |
| `partially-outdated` | частично устарел; что именно — отмечено внутри; кандидат на amend |
| `superseded` | замещён другим документом; только история |

## Что от вас потребуется

- **feature:** инициировать после мёржа; подтвердить, какие чужие доки получили `partially-outdated`/`superseded`
- **backfill:** выбрать эталонный код, прогнать survey, инициировать промоушен; подтвердить уровень (паттерн или экземпляр) и Area
