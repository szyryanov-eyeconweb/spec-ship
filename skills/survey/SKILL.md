# Survey

Трассирует существующий код от одного конкретного якоря и производит артефакт `Survey v1` JSON — доказательную карту связанных поверхностей. Только наблюдение: НЕ планирует новое поведение, НЕ меняет файлы.

Потребители survey: shape-doc (контекст вместо холодного интервью), decompose (`files_to_change`/`files_read_only` из `files_evidence`, не из головы), build (старт шейпа LOGIC от `observed_workflows`).

## Когда запускать

- фича меняет существующее поведение (не greenfield)
- изменение нелокальное: задевает persistence, propagation, несколько слоёв
- рефакторинг с сохранением поведения
- один caller сломан из-за смены допущения (идентификатор, ownership, payload)

Greenfield без якоря в существующем коде → пропустить, идти сразу в shape-doc.

## Процесс

### 1. Зафиксировать якорь

Записать:
- точный entrypoint: caller, символ, роут, GraphQL-поле, consumer, команда, падающий тест
- изменяемое допущение: что именно устаревает (контракт, scope идентификатора, форма payload, правило персистентности)
- почему изменение нелокальное

Если якорь неоднозначен — спросить пользователя, не угадывать.

### 2. Собрать контекст

- Прочитать `CONTEXT.md` (доменный глоссарий — имена состояний в workflow должны совпадать).
- Прочитать `.ship/docs/adr/INDEX.md` (НЕ все ADR). Отфильтровать `Status: Accepted` где `Area` пересекает якорь, загрузить тела ТОЛЬКО matched.
- Прочитать `.ship/docs/workflows/INDEX.md`, если есть (НЕ все доки). По области якоря загрузить ТОЛЬКО matched со `Status: current` — это уже готовые `observed_workflows`: переиспользовать, трассировать только расхождения с кодом и не покрытые код-пути. `partially-outdated` — лишь как подсказку, перепроверять по коду.
- Проверить `.ship/pipeline/` на survey/артефакты по той же области — не трассировать заново то, что уже описано и не устарело.

### 3. Трассировать по ответственности

От якоря вверх (кто вызывает) и вниз (что вызывается). Группировать НЕ по файлам — по ответственности:

1. **Entry и оркестрация** — роуты, GraphQL-резолверы, консольные команды, Kafka/RabbitMQ-консьюмеры, Temporal-воркфлоу
2. **Загрузчики и резолверы** — хелперы, query-билдеры, репозитории
3. **Persistence** — write-хелперы, Doctrine-модели, таблицы PG, таблицы/MV ClickHouse, миграции
4. **Ответ и propagation** — API-ответы, события Kafka, кеш Redis, проекции
5. **Потребители** — внешние читатели контракта: GraphQL-клиенты, downstream-консьюмеры, отчёты
6. **Validation** — unit/functional/acceptance тесты, фикстуры Behat, статические поверхности

Для каждой связанной части: зачем она здесь (связь с якорем), точные символы `путь#метод`, риск если пропустить.

Пустая группа — нормально, но отметить что проверена.

### 4. Записать наблюдаемые workflow

Для каждого затронутого код-пути — компактная workflow-строка от триггера до конечного состояния:

```
trigger --шаг--> состояние 1 --шаг--> состояние 2
[ветка 1, ветка 2]        # условные ветки
(ветка 1, ветка 2)        # параллельные ветки
```

Канон нотации — README («Workflow-нотация»). Имена состояний — из доменного глоссария `CONTEXT.md`. Короткие шаги, ориентированные на действие. Workflow описывает код КАК ЕСТЬ, не как должно стать — рефакторинг-форма `{...} --> {...}` здесь не используется (она для TO-BE в shape-doc).

### 5. Зафиксировать границы валидации

Где слабо типизированный/внешний вход входит в workflow, где валидируется/нормализуется, на какой пост-валидационный контракт полагаются внутренние шаги. Где существует отдельная граница доверия, требующая ре-валидации.

### 6. Собрать files_evidence

Для каждого файла — путь И причина:
- `to_change` — файлы, которые придётся менять при изменении допущения из якоря
- `read_only` — файлы, которые надо читать/учитывать, но не менять

Причина обязательна. «Найден грепом» — не причина; причина — роль файла в связанной группе.

### 7. Сохранить артефакт

- BusinessDoc уже существует → `.ship/pipeline/{slug}/survey-<bd-id>.json`
- BusinessDoc ещё нет (survey до shape) → `.ship/pipeline/_intake/survey-<YYYY>-<NNNN>.json`; shape-doc перенесёт в slug-директорию при создании BusinessDoc

Показать пользователю: якорь, размер карты (групп/символов), найденные риски, пустые группы.

## Схема выхода Survey

```jsonc
{
  "$schema": "pipeline/survey/v1",
  "id": "survey-2026-0007",
  "created_at": "<ISO8601>",
  "created_by": "dev:<username>",
  "business_doc_id": null,          // bd-id если survey делается для существующего BusinessDoc

  "anchor": {
    "entrypoint": "<точный символ/роут/тест>",
    "changed_assumption": "<что устаревает>",
    "why_non_local": "<почему не локальная правка>"
  },

  "observed_workflows": [
    {
      "name": "<имя код-пути>",
      "workflow": "trigger --шаг--> состояние --шаг--> [ветка 1, ветка 2]",
      "entry_symbol": "<путь>#<метод>"
    }
  ],

  "connected_groups": [
    {
      "group": "persistence",       // entry_orchestration | loaders_resolvers | persistence |
                                    // response_propagation | consumers | validation
      "symbols": ["src/<путь>.php#<метод>"],
      "why_connected": "<связь с якорем>",
      "risk_if_skipped": "<что сломается при пропуске>"
    }
  ],

  "validation_boundaries": [
    {
      "where": "<символ/слой входа слабого инпута>",
      "validates": "<что проверяется/нормализуется>",
      "post_validation_contract": "<на что могут полагаться внутренние шаги>",
      "revalidation_required_at": null   // или "<граница доверия>"
    }
  ],

  "files_evidence": {
    "to_change": [
      { "path": "<путь>", "reason": "<роль в изменении>" }
    ],
    "read_only": [
      { "path": "<путь>", "reason": "<зачем учитывать>" }
    ]
  },

  "adr_refs": ["adr-007"],          // Accepted ADR, пересекающие область якоря

  "empty_groups_checked": ["consumers"]  // проверенные и пустые группы
}
```

## Правила

- Survey — наблюдение. Ноль правок в `src/`, `tests/`, `config/`. Единственная запись — сам артефакт.
- Не планировать новое поведение: `observed_workflows` описывают код как есть. Планирование — shape-doc/decompose.
- Каждый файл в `files_evidence` имеет причину. Каждая связанная группа имеет `why_connected` и `risk_if_skipped`.
- Группы по ответственности, не по результатам грепа. «Файл нашёлся поиском» ≠ «файл связан».
- Имена состояний workflow — из `CONTEXT.md`, не выдуманные синонимы.
- Сохранить файл до отчёта. Никогда не выводить JSON только в чат.

## Типовые провалы (избегать)

- список файлов без объяснения связи
- workflow верхнего уровня без читателей/писателей за ним
- «frontend затронут» без имени конкретного потребителя контракта
- пропуск propagation: поменяли persistence, забыли события/проекции/кеш
- трассировка только вниз от якоря (забыли вызывающих)
