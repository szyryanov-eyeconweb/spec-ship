# Doc Backfill

Обёртка для наполнения канона поведением **существующего** кода — до того как по этой области пойдёт новая фича. Отвечает за «когда» (заранее, без гейта) и «откуда» (survey эталона). Конвертацию в документ делает `doc-promote-internal` — здесь её нет.

Цель: типовые паттерны описать один раз, чтобы будущий survey/shape-doc читал готовый канон, а не трассировал заново.

## Чем отличается от doc-promote-feature

| | doc-promote-feature | doc-backfill |
|---|---|---|
| Когда | после мёржа фичи | заранее, из готового кода |
| Гейт | APPROVED + MR смёржен | НЕТ (фичи нет) |
| Источник | survey + bd | только survey |
| Source-помета | `<bd-id>, <survey-id>` | `backfill (<survey-id>)` |

## Предусловие

- [ ] есть `survey-*.json`, трассирующий код ТЕКУЩЕЙ ветки

Нет survey → сначала прогнать `/spec-ship:survey` по эталонному символу. MR-гейт и ReviewReport НЕ требуются.

## Процесс

### 1. Выбрать эталон и получить survey

Эталон — репрезентативный пример паттерна (один graphql-эндпоинт, один отчёт, один консьюмер). Если survey ещё нет — запустить `survey` по якорю-эталону; он ляжет в `.ship/pipeline/_intake/survey-*.json`.

### 2. Решить уровень документа

- **паттерн** — шаблон слоёв, переиспользуемый для будущих фич (имена обобщены, напр. `{Report}`); Area по слою (`api`, `importer`, ...)
- **экземпляр** — конкретный код-путь как есть

Паттерн и экземпляр могут сосуществовать и ссылаться друг на друга (паттерн = шаблон, экземпляр = его реализация). Подтвердить уровень и Area у пользователя, если неоднозначно.

### 3. Вызвать конвертер

Вызвать `doc-promote-internal` через Skill tool, передав:
- source-артефакты: ТОЛЬКО survey (bd нет)
- Source-помета: `backfill (<survey-id>)`
- намерение по слою bd: bd отсутствует → итог КАК ЕСТЬ (observed_workflows survey переносятся напрямую, без слоя bd)

### 4. Подтвердить

Показать результат конвертера: путь документа, строки INDEX. Отметить, что Source — `backfill` (канон из готового кода, не из пройденной пайплайн-фичи).

## Жизненный цикл backfill-документа

Backfill-док помечен в Source как `backfill`. Когда фича позже изменит этот код-путь, `doc-promote-feature` сделает amend и заместит backfill-наблюдение точным TO-BE. До тех пор это авторитетное `current`-описание существующего поведения.

## Правила

- Гейта нет by design — backfill описывает код, который РЕАЛЬНО есть сейчас. Но survey обязан трассировать ТЕКУЩУЮ ветку: backfill с чужой/неактуальной ветки запишет в канон ложь.
- Конвертация — в `doc-promote-internal`; здесь только выбор эталона, survey и помета.
- Для поведения смёрженной фичи — другой сценарий, скилл `doc-promote-feature`.
