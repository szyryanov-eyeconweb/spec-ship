spec-ship — Spec-to-Ship Pipeline v2

Survey → Shape → Decompose → Build (Two-Agent TDD) → Review → Delivery. Артефакты в .ship/pipeline/{slug}/, два канона знаний: .ship/docs/adr/ (решения) и .ship/docs/workflows/ (поведение).

/spec-ship:survey
Survey Agent

  • Трассировка вверх и вниз по 6 группам ответственности: entry/оркестрация, загрузчики, persistence, propagation, потребители, validation
  • Наблюдаемые workflow в стрелочной нотации, имена состояний из CONTEXT.md
  • Границы валидации + пост-валидационные контракты
  • files_evidence: каждый файл с причиной («найден грепом» — не причина)

/spec-ship:shape-doc
Shape Agent
BA + агент (интерактивно)
  • Принимает сырую идею или тикет
  • Survey есть → вопросы не «как работает?», а «что должно измениться?»
  • Интервью: акторы, happy/sad/edge пути, DoD, constraints
  • Батчинг вопросов v2: не останавливаться на каждой неясности, копить связный батч; blocking | non_blocking (едут под явным assumption)
  • Ветвистые сценарии — стрелочный workflow-синтаксис в feature.workflow / ac.workflow
  • Гейт заморозки: blocking без resolution → НЕ апрувить
  • ADR-конфликт → СТОП, ADR-CONFLICT flow

1Decomposeавто
Агент разбивает фичу на вертикальные слайсы, классифицирует по зоне доверия формальными признаками.
Создаёт: task TaskSpec v2 ×N · conflict TestUpdateTicket (при конфликте теста)
Скиллы этапа
/spec-ship:decompose
Decompose Agent
Автономно + апрув разбивки BA
  • Гейт v2: blocking-вопрос без resolution → СТОП, не декомпозировать
  • Вертикальные слайсы: каждый задевает все слои, маппится на ≥1 AC
  • trust_zone формально v2: интерфейса хватает → ROUTINE (shape: null); нужны структуры/алгоритм → LOGIC (скелет shape: proposal); миграции/целостность → CRITICAL
  • files_to_change / files_read_only из survey files_evidence — доказательно, не из головы
  • ac.workflow → test_scenarios.workflow: ветки [...] дают отдельные сценарии
  • TestUpdateTicket при конфликте с существующими тестами
/spec-ship:decompose — декомпозиция + валидация покрытия AC + сохранение task-*.json

2Buildпараллельно / ночью
Три трека по зоне доверия. ROUTINE автономно. Шейп LOGIC персистится в TaskSpec.shape — не испаряется с сессией.
Создаёт: tests тесты (ship-red) · code реализацию (ship-green) · task-result BuildReport · adr ADREntry (кандидат)
Скиллы этапа
/spec-ship:build
Two-Agent TDD
ROUTINE — полностью автономно
Agent RED
Тест на каждый сценарий. v2: workflow/input/expected_outcome — контракт: arrange из input, assert ровно expected_outcome. Не видит src → тесты честные.
только tests/
Agent GREEN
Цель: все тесты зелёные. Не может трогать тесты. v2: docstring называет ts-id и шаг workflow (traceability).
только files_to_change
Эскалации v2: RED blocked / GREEN за N итераций → скелет shape (proposal, причина → open_for_developer) → трек LOGIC. GREEN конфликт → TestUpdateTicket, стоп.
/spec-ship:build — оркестратор: сабагенты ship-red + ship-green, изолированные права = физический барьер
/spec-ship:build — ADR Writer: hard-to-reverse решения → adr-entry-*.json (Proposed)
ctxtasktask-resultadrcodetests
Dev + Assistant Agent
LOGIC — нестандартные задачи
  • Шейп-сессия ~15 мин; повестка = shape.open_for_developer, контекст = survey.observed_workflows
  • Результат пишется в поле shape TaskSpec v2; Dev апрувит → status: approved
  • GREEN запускается только при approved, реализует по артефакту — не по пересказу
  • Сюда же эскалации из ROUTINE (скелет shape уже создан)

3Reviewасинхронно
Агент проверяет соответствие spec и ADR. Человек подключается только при ESCALATE.
Создаёт: review ReviewReport (APPROVED / NEEDS_WORK / ESCALATE) · гейт mr_ready
Скиллы этапа
/spec-ship:review
Review Agent
Автономно
  • Реализация соответствует spec.interface
  • Все test_scenarios покрыты; v2: тест ассертит ровно expected_outcome, не ослабленную версию
  • ADR не нарушены, нет ссылок на Expired
  • Нет регрессий в смежных модулях
  • Performance против DoD
✓ APPROVED → MR
↻ NEEDS_WORK → итерация
↑ ESCALATE → Dev
/spec-ship:review — 5-чеклист: spec_coverage, test_scenarios_covered, adr_violations, regressions, performance
ctxtaskadrtask-result
TEST-UPDATE flow
При устаревшем тесте
  • Обнаруживает: Decomposer, GREEN при конфликте, CI
  • Отдельный тикет, никогда inline-правка
  • Agent RED обновляет тест (НЕ GREEN), Review проверяет честность
  • pending-тикет → блок MR
/spec-ship:review — tu-*.json pending → blocking issue, mr_ready: false

Контексты карточек
surveysurvey-*.json — Survey — якорь, наблюдаемые workflow, связанные группы, files_evidence. Создаёт /spec-ship:survey (Phase 0a). Питает Shape, Decompose, Build (шейп LOGIC), Doc Promote.
docbd-*.json — BusinessDoc v2 — требования, AC (+workflow), DoD, open_questions. Создаёт /spec-ship:shape-doc (Phase 0). Замораживается после апрува BA; blocking-вопросы без resolution блокируют заморозку.
tasktask-*.json — TaskSpec v2 — интерфейс, файлы, test_scenarios (workflow/input/expected_outcome), trust_zone, shape. Создаёт /spec-ship:decompose (Phase 1). Поле shape — единственная разрешённая поздняя правка (шейп-сессия Build).
task-resultbuild-*.json — BuildReport — итерации TDD, эскалации, self-review. Создаёт /spec-ship:build (Phase 2). Читается /spec-ship:review на Phase 3.
adradr-entry-*.json — ADREntry — архитектурное решение (кандидат Proposed). Создаёт /spec-ship:build (Phase 2). Промоутится /spec-ship:adr-promote → .ship/docs/adr/ADR-NNN.md на Delivery.
wf-doc.ship/docs/workflows/*.md — канон поведения — Definitions, workflow-трассы, словарь состояний; INDEX со статусами current / partially-outdated / superseded. Создаёт /spec-ship:doc-promote-feature (Delivery, после мёржа) или /spec-ship:doc-backfill (заранее, из кода). Читается survey вместо повторной трассировки.
codesrc/ — исходный код — читается для понимания интерфейсов. Пишет только ship-green (Phase 2), и только files_to_change.
teststests/ — тесты — контракт между RED и GREEN. Пишет только ship-red. ship-green не трогает. Обновление — только через TestUpdateTicket + апрув.
conflicttu-*.json — TestUpdateTicket — конфликт существующего теста с новым spec. Создаёт decompose (Phase 1) или build (Phase 2). Разрешает Agent RED + апрув maintainer.
reviewreview-*.json — ReviewReport — вердикт APPROVED / NEEDS_WORK / ESCALATE + чеклист 5 проверок. Создаёт /spec-ship:review (Phase 3). Гейт mr_ready: true → MR.
ctxCONTEXT.md — доменный глоссарий — термины проекта, имена состояний workflow на всех уровнях. Читается на всех фазах. Обновляется на Phase 0.
