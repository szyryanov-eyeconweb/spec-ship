# Ship Doc Promote

Конвертит знание из одноразовых артефактов `.ship/pipeline/{slug}/` (Survey, BusinessDoc) в долгоживущие workflow-документы `.ship/docs/workflows/` и ведёт их жизненный цикл через `.ship/docs/workflows/INDEX.md`.

Зеркало adr-promote, но для другого вида знания: ADR хранит «почему решили так», workflow-док — «как система себя ведёт». Пайплайн-артефакты остаются одноразовым аудит-следом; канон поведения живёт здесь.

Это ручной шаг maintainer'а на Delivery — НЕ автономный. Агент исполняет конвертацию, человек инициирует и апрувит.

## Предусловия

- [ ] `ReviewReport.verdict == "APPROVED"` для задач фичи
- [ ] MR смёржен — документ описывает поведение, которое РЕАЛЬНО в ветке, не планируемое

Если любое не выполнено — НЕ промоутить, surface причину.

## Источники

Из `.ship/pipeline/{slug}/`:
- `survey-*.json` → `observed_workflows` (поведение ДО фичи), `connected_groups`, `validation_boundaries`
- `bd-*.json` → `feature.workflow`, `acceptance_criteria[].workflow` (поведение ПОСЛЕ фичи), акторы, constraints, `data[]` (управляющие значения — в Definitions, если поведение без них непонятно; большие матрицы — выдержкой с ссылкой на артефакт)
- `task-*.json` → `shape` (алгоритмические структуры — если поведение без них непонятно), `spec.files_to_change` (актуальные символы)

Итоговый документ описывает состояние ПОСЛЕ мёржа: workflow из bd замещают изменённые ими workflow из survey, не затронутые фичей наблюдения из survey переносятся как есть.

## Процесс

### 1. Определить целевой документ

Вывести `Area` из затронутых файлов по feature-based структуре (та же логика, что в adr-promote): `projection`, `importer`, `api`, `reports`, `shared`, `architecture`.

Прочитать `.ship/docs/workflows/INDEX.md` (создать при первом промоушене):
- документ по этой области/код-пути уже есть → режим **amend** (обновить его)
- нет → режим **create** (`.ship/docs/workflows/<area>-<kebab-имя-код-пути>.md`)

Если фича изменила поведение, описанное и в ДРУГИХ документах INDEX, — для каждого решить: amend | `partially-outdated` | `superseded`. Не оставлять молча лгущий документ.

### 2. Собрать/обновить документ

Формат (выдержки достаточны, не раздувать):

```markdown
# <Имя код-пути или области>

**Status:** current
**Updated:** <ISO-date>
**Area:** <area>
**Source:** <bd-id>, <survey-id>   // последние повлиявшие артефакты

<Назначение и охват в 1–2 предложениях.>

## Definitions

### <Имя структуры/состояния>
Type: <runtime type | таблица PG | таблица/MV ClickHouse | событие Kafka>
Purpose: <роль в workflow>
Key fields: <важные поля>

## Workflows

### <Имя workflow>
`trigger --шаг--> состояние 1 --шаг--> [ветка 1, ветка 2]`

<Пояснения: границы валидации и пост-валидационные контракты,
неочевидные инварианты. Символы кода: путь#метод.>

## State Name Dictionary

| Состояние в workflow | Definition |
| --- | --- |
```

Правила содержания:
- стрелочная нотация та же, что в survey/bd/task (канон — README «Workflow-нотация») — копировать строки из артефактов, не пересочинять
- если `feature.workflow` в bd записан рефакторинг-формой `{старый} --шаг--> {новый}` — левая часть называет замещаемый workflow канона, в документ пишется правая часть (новое поведение); сама рефакторинг-строка в канон не копируется
- имена состояний из `CONTEXT.md`; каждое состояние workflow имеет строку в Definitions или Dictionary
- документ читается без открытия кода
- при amend: обновить изменённые workflow, `Updated`, дописать `Source`; не переписывать незатронутое

### 3. Зарегистрировать в INDEX

`.ship/docs/workflows/INDEX.md` — таблица:

```markdown
| Document | Status | Area | Source | Updated |
| --- | --- | --- | --- | --- |
| importer-money-flow-dedup.md | current | importer | bd-2026-0002 | 2026-06-11 |
```

- create → новая строка `current`
- amend → обновить `Source`/`Updated`, статус остаётся `current`
- задетые чужие документы → проставить `partially-outdated` / `superseded` (строку не удалять)

### 4. Подтвердить

Показать: путь документа (create/amend), изменённые строки INDEX, какие чужие документы получили `partially-outdated`/`superseded`.

## Жизненный цикл документов

| Status | Значение | Кто читает |
|---|---|---|
| `current` | авторитетно описывает поведение в ветке | survey (вместо повторной трассировки), onboarding, Dev |
| `partially-outdated` | частично устарел после фичи; что именно — отмечено в документе | читать с осторожностью; кандидат на amend следующим промоушеном этой области |
| `superseded` | замещён другим документом (ссылка в строке INDEX) | только история |

Потребитель `current`-документов — survey (шаг «Собрать контекст»): по области якоря сперва INDEX, тела ТОЛЬКО matched `current` — трассировать заново только расхождения и не покрытое. Тот же паттерн выборки, что у ADR через INDEX.

## Правила

- НЕ промоутить до мёржа MR. Канон описывает реальность, не намерение.
- Документ — про поведение («как работает»), не про решения («почему так») — для решений есть ADR. Не дублировать ADR-контент; при необходимости ссылка `см. ADR-NNN`.
- INDEX — единственная точка выборки. Скиллы не сканируют `.ship/docs/workflows/*.md` подряд.
- Статусы только через INDEX + шапку документа; устаревший документ не удаляется.
- Выдержки, не энциклопедия: документ покрывает код-пути, затронутые хотя бы одной фичей пайплайна. Рост корпуса сдержан областями и amend-режимом.
