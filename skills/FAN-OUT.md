# FAN-OUT — параллель build по слоям

Сквозной под-трек. Ортогонален `trust_zone` — ускоряет ROUTINE и LOGIC параллелью по слоям-ролям внутри ОДНОЙ задачи. Применяют `decompose` (ставит `fan_out` в TaskSpec) и `build` (исполняет Phase A/A.5/B/C). Канон механики — здесь; build/decompose ссылаются, не дублируют.

**Это оптимизация скорости, не дефолт.** Обычная задача — последовательный build, `fan_out: null`. Читать этот файл только когда задача реально фан-аут-able (см. «Когда ставить»).

Слои зависят не от кода друг друга, а от **контракта** (порты + DTO). Зафиксировали контракт в Phase A → реализации слоёв независимы → Phase B параллельна. Пересечение возможно ТОЛЬКО при незрелом контракте.

## Слои — по РОЛИ, не по структуре проекта

Пути проектные, роли универсальны:
- `entry` — приём извне: HTTP/GraphQL-резолверы, очередь-консьюмеры, RPC-хендлеры, workflow-входы;
- `application` — use-case, Handler, ПОРТЫ (интерфейсы) и DTO; здесь рождается контракт, который реализуют остальные;
- `contract-impl` — реализации портов application (`implements <PortИзApplication>`): запросы к БД, репозитории, внешние клиенты.

## Когда ставить `fan_out.enabled: true` (decompose)

ТОЛЬКО когда ВСЕ три:
1. задача делится на ≥2 слоя-роли с непересекающимися путями;
2. контракт (порты + DTO) фиксируем ДО реализации — известен из `spec.interface`, `shape` или паттерн-дока `.ship/docs/workflows/`;
3. `trust_zone != CRITICAL`.

Окупается при ≥3 нетривиальных слоях и заранее известном контракте (есть паттерн-канон). Сомнение в зрелости контракта → обычный последовательный build.

### Поля fan_out

- `contract_paths` — артефакты контракта (порты + DTO): Phase A пишет, дальше read-only для слоёв.
- `shared_paths` — **общая земля**: файлы, которые трогают НЕСКОЛЬКО слоёв, но не контракт — DI-реестр (`services.yaml`), регистрация сервисов/маршрутов, явные записи схемы. Нельзя нарезать по слою (каждый дописывает в один файл → конфликт). Phase A пишет их целиком за все слои (регистрация вперёд реализаций). `[]` если нет. Генерируемую землю (схема из аннотаций, DI-кеш) сюда НЕ класть — регенерируется в Phase C.
- `layers[]` — `{ role, files_to_change }` на слой.

### Инварианты (decompose валидирует перед `enabled: true`)

1. `files_to_change` слоёв попарно НЕ пересекаются;
2. `contract_paths` ∩ любой `layer.files_to_change` = пусто;
3. `shared_paths` ∩ любой `layer.files_to_change` = пусто.

Также: `layers[].files_to_change ∪ contract_paths ∪ shared_paths` покрывает весь `spec.files_to_change`; `contract_paths` и `shared_paths` ⊆ `spec.files_to_change`. Нарушение любого → не fan_out, обычный build.

## Фазы build

### Phase A — Contract + общая земля (последовательно, сессия)

Написать контракт-артефакты из `contract_paths`: порты-интерфейсы application + DTO (сигнатуры, поля, типы), без тел реализаций. Источник скелета:
- ROUTINE+fan_out: `spec.interface` + паттерн-док из `.ship/docs/workflows/` (если по области есть — готовый список портов/DTO, не изобретать);
- LOGIC+fan_out: `shape.intermediate_structures` (имена + инварианты = контракт), только после `shape.status: approved`.

ЗАТЕМ написать общую землю из `shared_paths` (если непусто) — ЦЕЛИКОМ за ВСЕ слои сразу, включая регистрацию ещё-не-реализованных классов. Снимает конфликт «каждый слой дописывает в один файл». Генерируемую землю в Phase A НЕ трогать.

Контракт + общая земля заморожены → для Phase B read-only.

### Phase A.5 — RED (последовательно)

Вызвать `ship-red`: ВСЕ тесты задачи по `test_scenarios`, через публичный интерфейс, без моков доменных внутренностей. Тесты ассертят сквозное поведение (стык слоёв), не отдельный слой. Подтвердить ALL RED. Тесты — общий контракт поведения для всех слоёв. Исход `blocked` → как в ROUTINE-треке (переклассификация, стоп).

### Phase B — Fan-out (параллельно, каждый слой в своём worktree)

По одному `ship-green` на слой `layers[]`, параллельно, **каждый в отдельном git worktree** (`isolation: 'worktree'`). Каждому передать: TaskSpec, `contract_paths` + `shared_paths` (read-only), список красных тест-классов, его `layer.files_to_change` как границу записи.

Зачем worktree (поверх инварианта непересечения):
- физический барьер ФС — слой не затрёт чужой файл даже по ошибке (страховка инварианта 1, не замена);
- детектор вылазок — после слоя проверить diff worktree: запись вне `layer.files_to_change` (за вычетом read-only contract/shared) = **незамеченная общая земля** → стоп, вернуть в decompose дополнить `shared_paths`, дописать землю в Phase A, перезапустить B.

Семантический рассинхрон слоёв (контракт неточен) worktree НЕ ловит — это Phase C.

Обработка исходов по каждому слою — как в ROUTINE-треке (`done` → ок; `escalated`(N)/`conflict`/ADR-сигнал → соответствующий flow). ДОПОЛНИТЕЛЬНО:
- **незрелый контракт**: слой сообщает «порту/DTO не хватает X» → STOP ВСЕГО фан-аута (НЕ чинить контракт внутри Phase B — гонка: двое правят порт) → вернуться в Phase A, дополнить `contract_paths`, заморозить заново → перезапустить Phase B.
- **вылазка за границы** (детектор worktree): слой тронул файл вне границ → STOP → decompose дополняет `shared_paths` → Phase A дописывает землю → перезапуск B.

### Phase C — Merge + Integration (последовательно, сессия)

1. **Merge worktree-слоёв** в одну ветку. Слой-файлы непересекающиеся → union без конфликтов. `contract_paths`/`shared_paths` одинаковы во всех worktree (read-only) → тривиально. Конфликт merge здесь = вылазка, пропущенная детектором → стоп, в decompose.
2. **Регенерация генерируемой земли** из объединённого дерева: схема из аннотаций, DI-кеш, autoload. Не мержить — пересоздать командой проекта.
3. **Полный прогон suite.** Стык слоёв не сошёлся (слой вернул не то, что другой ждал) → контракт был неточен → вернуться в Phase A. Зелёно → self-review по полному результату.

## Риски и кто чинит

| Конфликт | Чинит |
|---|---|
| race одного файла | инвариант 1 (непересечение) + worktree как страховка |
| общая земля (config/DI/схема) | `shared_paths` в Phase A + worktree-детектор вылазок |
| незрелый контракт (семантика) | Phase C, рестарт Phase A |

## Выигрыш / цена

- Выигрыш: wall-clock Phase B = max(слой), не sum(слои). 3 слоя → до ~3× на этапе.
- Цена: последовательные Phase A + A.5 + риск рестарта при незрелом контракте.
- Окупается: ≥3 нетривиальных слоя + контракт известен заранее (канон есть).
- НЕ окупается: правка одного класса, 1–2 слоя, незрелый контракт.
