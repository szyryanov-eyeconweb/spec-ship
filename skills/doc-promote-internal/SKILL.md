# Doc Promote — внутренний конвертер

Чистый конвертер: артефакты поведения → workflow-канон. **Не знает «когда» и «откуда взялось»** — это забота вызывающих обёрток. Не проверяет гейты (APPROVED/MR), не ищет эталоны, не решает режим запуска. Делает ровно одно: собирает/обновляет workflow-документ и регистрирует его в INDEX.

## Контракт вызова

Вызывается через Skill tool **только** из:
- `doc-promote-feature` — после мёржа фичи (гейт там)
- `doc-backfill` — из survey существующего кода (без гейта)

Обёртка передаёт:
- **source-артефакты**: `survey-*.json` (обязателен) + опционально `bd-*.json`, `task-*.json`
- **Source-помета** для шапки документа: строка `<bd-id>, <survey-id>` или `backfill (<survey-id>)` — конвертер пишет что дали, не вычисляет
- **намерение по слою bd**: есть bd → итог описывает поведение ПОСЛЕ (workflow из bd замещают изменённые наблюдения survey, незатронутые переносятся); нет bd → итог КАК ЕСТЬ (observed_workflows survey напрямую)

Если обязательный survey не передан или нечитаем — STOP, сообщить вызывающему. Гейты конвертер НЕ проверяет — доверяет обёртке.

## Источники → что брать

- `survey-*.json` → `observed_workflows`, `connected_groups`, `validation_boundaries`
- `bd-*.json` (если передан) → `feature.workflow`, `acceptance_criteria[].workflow`, акторы, constraints, `data[]` (управляющие значения — в Definitions; большие матрицы — выдержкой со ссылкой на артефакт)
- `task-*.json` (если передан) → `shape` (алгоритмические структуры, если без них поведение непонятно), `spec.files_to_change` (актуальные символы)

## Процесс

### 1. Определить целевой документ

Вывести `Area` из затронутых файлов по feature-based структуре (та же логика, что в adr-promote): `projection`, `importer`, `api`, `reports`, `shared`, `architecture`.

Прочитать `.ship/docs/workflows/INDEX.md` (создать при первом промоушене):
- документ по этой области/код-пути уже есть → режим **amend** (обновить его)
- нет → режим **create** (`.ship/docs/workflows/<area>-<kebab-имя-код-пути>.md`)

Если переданное знание изменило поведение, описанное и в ДРУГИХ документах INDEX, — для каждого решить: amend | `partially-outdated` | `superseded`. Не оставлять молча лгущий документ.

### 2. Собрать/обновить документ

Формат (выдержки достаточны, не раздувать):

```markdown
# <Имя код-пути или области>

**Status:** current
**Updated:** <ISO-date>
**Area:** <area>
**Source:** <переданная обёрткой Source-помета>

<Назначение и охват в 1–2 предложениях.>

## Definitions

### <Имя структуры/состояния>
Type: <runtime type | таблица PG | таблица/MV ClickHouse | событие Kafka>
Purpose: <роль в workflow>
Key fields: <важные поля>

## Workflows

### <Имя workflow>
`trigger --шаг--> состояние 1 --шаг--> [ветка 1, ветка 2]`

<Пояснения: границы валидации и пост-валидационные контракты,
неочевидные инварианты. Символы кода: путь#метод.>

## State Name Dictionary

| Состояние в workflow | Definition |
| --- | --- |
```

Правила содержания:
- стрелочная нотация та же, что в survey/bd/task (канон — README «Workflow-нотация») — копировать строки из артефактов, не пересочинять
- если `feature.workflow` в bd записан рефакторинг-формой `{старый} --шаг--> {новый}` — левая часть называет замещаемый workflow канона, в документ пишется правая часть (новое поведение); сама рефакторинг-строка в канон не копируется
- имена состояний из `CONTEXT.md`; каждое состояние workflow имеет строку в Definitions или Dictionary
- документ читается без открытия кода
- при amend: обновить изменённые workflow, `Updated`, дописать `Source`; не переписывать незатронутое

### 3. Зарегистрировать в INDEX

`.ship/docs/workflows/INDEX.md` — таблица:

```markdown
| Document | Status | Area | Source | Updated |
| --- | --- | --- | --- | --- |
| importer-money-flow-dedup.md | current | importer | bd-2026-0002 | 2026-06-11 |
```

- create → новая строка `current`
- amend → обновить `Source`/`Updated`, статус остаётся `current`
- задетые чужие документы → проставить `partially-outdated` / `superseded` (строку не удалять)

### 4. Вернуть результат вызывающему

Сообщить обёртке: путь документа (create/amend), изменённые строки INDEX, какие чужие документы получили `partially-outdated`/`superseded`. Обёртка покажет это пользователю.

## Жизненный цикл документов

| Status | Значение | Кто читает |
|---|---|---|
| `current` | авторитетно описывает поведение в ветке | survey (вместо повторной трассировки), onboarding, Dev |
| `partially-outdated` | частично устарел; что именно — отмечено в документе | читать с осторожностью; кандидат на amend |
| `superseded` | замещён другим документом (ссылка в строке INDEX) | только история |

Потребитель `current`-документов — survey (шаг «Собрать контекст»): по области якоря сперва INDEX, тела ТОЛЬКО matched `current`. Тот же паттерн выборки, что у ADR через INDEX.

## Правила

- **Только конвертация.** Гейты, выбор эталона, режим запуска — НЕ здесь, у обёрток. Конвертер не знает, фича это или backfill — он знает лишь переданный набор артефактов и Source-помету.
- Документ — про поведение («как работает»), не про решения («почему так») — для решений есть ADR. Не дублировать ADR; при необходимости ссылка `см. ADR-NNN`.
- INDEX — единственная точка выборки. Скиллы не сканируют `.ship/docs/workflows/*.md` подряд.
- Статусы только через INDEX + шапку документа; устаревший документ не удаляется.
- Выдержки, не энциклопедия. Рост корпуса сдержан областями и amend-режимом.
