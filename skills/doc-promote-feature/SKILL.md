# Doc Promote — feature

Обёртка для промоушена поведения **смёрженной фичи** в канон. Отвечает за «когда» (после мёржа, с гейтом) и «откуда» (артефакты фичи). Саму конвертацию в документ делает `doc-promote-internal` — здесь её нет.

Зеркало adr-promote, но для другого знания: ADR хранит «почему решили так», workflow-док — «как система себя ведёт».

Это ручной шаг maintainer'а на Delivery — НЕ автономный. Человек инициирует и апрувит.

## Гейт (проверить ДО конвертации)

- [ ] `ReviewReport.verdict == "APPROVED"` для задач фичи
- [ ] MR смёржен — канон описывает поведение, которое РЕАЛЬНО в ветке, не планируемое

Любое не выполнено → НЕ промоутить, surface причину. Конвертер гейт не проверяет — это ответственность здесь.

## Процесс

### 1. Определить slug и собрать источники

Из `.ship/pipeline/{slug}/`:
- `survey-*.json` — поведение ДО фичи (если был survey-этап)
- `bd-*.json` — `feature.workflow`, критерии (поведение ПОСЛЕ), акторы, constraints, `data[]`
- `task-*.json` — `shape`, `spec.files_to_change`

### 2. Вызвать конвертер

Вызвать `doc-promote-internal` через Skill tool, передав:
- source-артефакты: survey + bd (+ task)
- Source-помета: `<bd-id>, <survey-id>`
- намерение по слою bd: bd передан → итог описывает поведение ПОСЛЕ (workflow из bd замещают изменённые наблюдения survey, незатронутые переносятся)

### 3. Подтвердить

Показать пользователю результат, вернутый конвертером: путь документа (create/amend), изменённые строки INDEX, какие чужие документы получили `partially-outdated`/`superseded`.

## Правила

- НЕ промоутить до мёржа MR. Канон описывает реальность, не намерение.
- Гейт — здесь; конвертация — в `doc-promote-internal`; не дублировать логику конвертера.
- Для промоушена существующего кода БЕЗ фичи (типовые паттерны заранее) — это другой сценарий, скилл `doc-backfill`.
