# Ship Run — оркестратор полного цикла

Прогоняет фичу через весь пайплайн одним вызовом человека. Сам не делает работу этапов — последовательно вызывает скиллы `survey`, `shape-doc`, `decompose`, `build`, `review` и управляет переходами между ними. Цель — убрать рутину «5 ручных вызовов по очереди», сохранив ВСЕ гейты участия человека.

Этот скилл — тонкий дирижёр. Вся механика, схемы, эскалации — в скиллах этапов и в [README](../README.md). `run` их не дублирует, только маршрутизирует.

## Главный принцип

**Останавливайся только на блокирующих гейтах. Между ними — кати без пауз.**

Расходы пайплайна (токены, время) неизбежны — этапы всё равно выполняются. `run` экономит ДРУГОЕ: число ручных вызовов и переключений контекста человека. Человек подключается там, где решение действительно его, и не трогает то, что агент делает сам.

## Вход

Свободное описание фичи. Опционально — якорь в коде (если меняется существующее поведение) и флаги (см. ниже).

```
/spec-ship:run Фильтрация транзакций по меткам; якорь TransactionRepository#findByPartner
```

## Флаги автономности (опционально)

По умолчанию все гейты включены. Флаги снимают НИЗКОРИСКОВЫЕ гейты, остальные не отключаются никогда:

- `--auto-shape` — авто-апрув BusinessDoc, ЕСЛИ в нём 0 `open_questions` со `severity: blocking`. Иначе всё равно стоп.
- `--auto-decompose` — авто-апрув разбивки, ЕСЛИ ВСЕ слайсы `ROUTINE`. Любая LOGIC/CRITICAL → стоп на апрув.
- `--auto` — оба выше.

Гейты, которые флаги НЕ снимают никогда: blocking-вопросы интервью, шейп LOGIC, CRITICAL-задача, любые эскалации, review ESCALATE. Это точки, где решение принципиально человеческое.

## Поток

```
[0a] survey      — ТОЛЬКО если есть якорь / меняется существующее поведение. Автономно.
   ▼
[0]  shape-doc   — интервью. ГЕЙТ: blocking-вопросы → стоп. ГЕЙТ: апрув bd (если не --auto-shape с 0 blocking).
   ▼
[1]  decompose   — нарезка + trust_zone. ГЕЙТ: апрув разбивки (если не --auto-decompose со всеми ROUTINE).
   ▼
[2]  build ×N    — по каждой задаче в порядке depends_on:
        ROUTINE  → автономно (RED→GREEN), НЕ останавливаться
        LOGIC    → ГЕЙТ: шейп-сессия + апрув shape
        CRITICAL → ГЕЙТ: консультация, Dev пишет сам
        эскалация (RED blocked / GREEN N / tu / adr-conflict) → ГЕЙТ по протоколу этапа
   ▼
[3]  review ×N   — по каждому build:
        APPROVED   → автономно дальше
        NEEDS_WORK → агент чинит сам, повторное review, НЕ дёргать человека
        ESCALATE   → ГЕЙТ: стоп, к Dev с контекстом
   ▼
итог: сводка — что прошло APPROVED, что ждёт человека, что заэскалировано
```

## Процедура

### 1. Классифицировать запрос

- Есть якорь в существующем коде ИЛИ фича меняет поведение → начать с `survey`.
- Greenfield без якоря → пропустить survey, сразу `shape-doc`.
- Распарсить флаги автономности из аргументов.

### 2. Survey (если нужен)

Вызвать скилл `survey`. Автономно — survey только наблюдает, ничего не меняет. Артефакт `survey-*.json` ляжет в `_intake/` (bd ещё нет) и будет перенесён в slug на shape.

Переход к shape-doc без паузы.

### 3. Shape-doc

Вызвать скилл `shape-doc`. Он ведёт интервью (контекст из survey, если был).

**ГЕЙТ — blocking-вопросы:** если интервью упёрлось в `severity: blocking` без resolution — стоп, вернуть вопросы человеку. Это не обходится флагами.

**ГЕЙТ — апрув bd:**
- без `--auto-shape` → показать BusinessDoc, ждать явного «апрув».
- с `--auto-shape` → проверить `open_questions`: 0 blocking без resolution → авто-`approved`, продолжить; иначе стоп на апрув.

После approved → decompose без паузы.

### 4. Decompose

Вызвать скилл `decompose`. Он режет на TaskSpec и ставит `trust_zone`.

**ГЕЙТ — апрув разбивки:**
- без `--auto-decompose` → показать разбивку (title, trust_zone, depends_on), ждать «апрув».
- с `--auto-decompose` → если ВСЕ задачи `ROUTINE` → авто-апрув, продолжить; если есть LOGIC/CRITICAL → стоп на апрув (человек должен видеть, что часть работы потребует его участия).

После апрува bd → `frozen`, переход к build-циклу.

### 5. Build-цикл (по каждой задаче)

Определить порядок по `dependencies.depends_on` (топологически: задача не строится раньше тех, от кого зависит). Для каждой задачи вызвать скилл `build`:

- **ROUTINE** → build гоняет RED→GREEN автономно. Успех → сразу к review этой задачи (или копить и ревьюить пачкой — см. шаг 6). Человека НЕ дёргать.
- **LOGIC** → **ГЕЙТ:** провести шейп-сессию (это суть LOGIC, не обходится). После апрува shape build реализует. Затем review.
- **CRITICAL** → **ГЕЙТ:** консультация, Dev пишет код сам.
- **Эскалация** (RED `blocked`, GREEN `escalated` после N, `conflict`/tu, ADR-конфликт) → **ГЕЙТ** по протоколу соответствующего этапа (см. README «Эскалации»). Стоп по этой задаче, НО продолжить независимые задачи, которые не зависят от заблокированной (см. правило ниже).

### 6. Review-цикл

По каждому готовому build вызвать скилл `review`:

- **APPROVED** → отметить задачу готовой, продолжить.
- **NEEDS_WORK** → исправимо агентом: вернуть в build на починку, повторить review. Человека НЕ дёргать (это автономный цикл починки). Разумный лимит повторов — 2; дальше эскалировать как ESCALATE.
- **ESCALATE** → **ГЕЙТ:** стоп по задаче, поднять Dev с полным контекстом из ReviewReport.

### 7. Итоговая сводка

Когда все задачи прошли или упёрлись в гейты — показать сводку:
- задачи APPROVED (готовы к MR)
- задачи, ждущие человека: какой гейт, что нужно сделать
- заэскалированные: причина, контекст
- следующий шаг для человека (ответить на вопрос / апрувнуть / зашейпить / разобрать эскалацию), и каким `/spec-ship:run --resume` или точечной командой продолжить

## Правило независимости задач

Заблокированная задача (гейт/эскалация) НЕ должна стопить весь прогон. Продолжай задачи, не зависящие от неё по `depends_on`. Стоп всего прогона — только если заблокирован общий префикс (нет ни одной задачи, готовой к работе).

Это и есть выигрыш `run`: пока одна LOGIC-задача ждёт твоего шейпа, три ROUTINE уже собрались и отревьюились.

## Возобновление

`/spec-ship:run --resume` (или просто повторный `run` по той же фиче): прочитать состояние из `.ship/pipeline/{slug}/` — какие артефакты уже есть, на каком гейте остановились — и продолжить с первой незавершённой точки. Идемпотентно: готовые этапы не переделывать.

## Что НЕ делает run

- НЕ снимает гейты, кроме явно разрешённых флагами низкого риска.
- НЕ пишет код сам — вся работа в скиллах этапов и сабагентах.
- НЕ трогает Delivery-этапы (`adr-promote`, `doc-promote`) — они после мёржа MR, вне автопрогона, ручной шаг maintainer'а.
- НЕ заменяет точечные команды: одиночную задачу дешевле запустить прямым `/spec-ship:build`, чем гонять весь `run`.

## Когда использовать run vs точечные команды

| Ситуация | Что брать |
|---|---|
| фича на 3–15 слайсов, хочется один запуск | `run` |
| ночной автопрогон рутины, Telegram на гейтах | `run --auto` |
| одиночная понятная задача | прямой `/spec-ship:build` |
| только разведка / только спека | точечный `survey` / `shape-doc` |
| правка опечатки, бамп версии | без пайплайна |

## Правила

- Гейты участия человека приоритетнее автономности. Сомнение «гейт или нет» → гейт.
- Между автономными этапами и ROUTINE-задачами — ноль пауз.
- Каждый этап выполняет СВОЙ скилл; `run` только вызывает и маршрутизирует исходы.
- Состояние всегда в артефактах `.ship/pipeline/{slug}/`, не в памяти сессии — прогон возобновляем.
- Итоговая сводка обязательна: человек должен из одного сообщения понять, что готово и что ждёт его.
